# 需求澄清 SOP（Grill）

## 追问原则

1. **分轮追问**：每轮最多 1-2 个关键问题，不要一次抛一长串淹没用户。
2. **优先问影响架构与排期的**：P0 范围、性能/并发量级、安全合规、与已有框架的集成方式。
3. **给选项式确认**：尽量提供候选项让用户勾选/确认，而非纯开放式提问，降低用户成本。
4. **先查后问**：先读 `${CLAUDE_PLUGIN_ROOT}/rules/existing-framework.md` 与历史需求文档，框架已提供的能力不再追问，避免把已有能力当新需求。

## 追问维度（checklist）

| 维度 | 要问清的 |
|---|---|
| 业务 | 涉及哪些角色？权限边界？正常/异常流程？ |
| 数据 | 核心实体？关键字段？数据量级/增长预期？ |
| 非功能 | 并发/性能指标？安全/合规要求？可用性目标？ |
| 集成 | 与已有登录/系统管理的关系？依赖的外部系统？ |

## 结论沉淀（文件化持久）

追问发生在对话里，但结论必须落文件——上下文压缩可能压掉澄清细节：

- 澄清笔记路径：`docs/requirements/<feature-slug>-clarification-notes.md`（不存在则创建，头部写 feature 与开始日期）。
- **每轮追问结束即增量写入**：轮次 / 所提问题 / 用户回答 / 达成结论 / 本轮遗留项。只追加，不重写历史轮次。
- 未取值的非功能性要求在笔记中标注「待确认」——**禁止编造**。

## 收敛标准（满足才可定稿）

- 每个 `REQ-xxx` 都有可转化为测试断言的验收标准。
- 非功能性要求要么有明确取值，要么显式标注「待确认」——**禁止编造具体数字**。
- 与已有框架能力、历史需求无未解决冲突。
- 未决问题已显式列出（编排模式追加交接块时写入其 `open_questions`，见 `${CLAUDE_PLUGIN_ROOT}/skills/context-handoff/SKILL.md` 定稿协议）。
