# rules/ 目录说明

本目录存放**技术栈相关的编码/接口规约**，供 `agents/architect.md` 与 `agents/developer.md` 按需加载，不放业务逻辑。

## 现有规则

| 文件 | 覆盖范围 |
|---|---|
| java.md | Java 后端通用规范：命名、分层、异常处理、日志规范 |
| spring.md | Spring / Spring Boot 专项：统一响应体、全局异常处理、配置约定、接口路径规范 |
| vue.md | Vue 前端规范：组件结构、状态管理、API 封装、命名约定 |
| existing-framework.md | 已有基础框架（登录鉴权、系统管理、菜单权限等）能力清单与扩展规范，**所有阶段必读** |

## 新增技术栈规则的步骤

1. 在本目录新增 `<stack>.md`（如 `go.md`、`react.md`、`uniapp.md`）。
2. 在 `skills/pipeline-overview/SKILL.md` 第 4 节"技术栈规则路由表"新增一行，说明该规则文件的适用场景。
3. 规则文件建议保持统一结构，便于 agent 稳定解析、也便于人工维护：
   - 命名规范
   - 分层/目录结构约定
   - 通用组件/工具的使用方式（如统一响应体、统一异常）
   - 反模式（禁止事项）
   - 示例代码片段（对应 `templates/code/<stack>/` 下的模板）

## 编写原则

规则文件应尽量精简、可执行，**每条规则最好能直接对应一条代码评审(code review) checklist 项**，避免写成"最佳实践论文"——那样会被模型当作背景知识略读，而不是当作强约束遵守。
